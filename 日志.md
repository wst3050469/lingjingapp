## v1.73.80 升级稳定性增强 (2026-06-15)

### 问题背景
1. **安装包文件未找到**: 用户升级到 v1.73.79 时偶尔报错"安装包文件未找到"
2. **lingjing-update-server 频繁崩溃**: 每~40秒重启一次 (83次/56分钟)

### 根因分析

#### 安装包未找到
- `/api/latest` 通过 nginx → cloud-server(:8000) 正常返回 v1.73.79 + status:published
- 所有安装包文件存在且可下载 (HTTP 200)
- 真正原因: **cloud-server 瞬断** + **lingjing-update-server 崩溃循环** 导致 `/api/latest` 偶发不可达
- 旧版客户端 HTTP fallback 无重试机制，一次失败即报错

#### lingjing-update-server 崩溃循环
- 每次请求都同步读取/解析 versions.json（无缓存）
- 无 `process.on('uncaughtException')` 处理
- 无请求级 try-catch 保护

### 修复内容

| 文件 | 修改 | 说明 |
|------|------|------|
| `services/update/lingjing-update-server.js` | 重构 | v1.1.0: 60s内存缓存 + uncaughtException防护 + 请求级try-catch |
| `desktop/electron/src/ipc/update-ipc.ts` | 增强 | URL预探测 + HTTP fallback 5次重试 + 版本状态重检 |
| `infra/scripts/check-release-ready.sh` | 新增 | 版本发布就绪验证脚本 (API/安装包/服务健康) |

### 修复详情

#### lingjing-update-server.js (v1.1.0)
- `getVersions()`: 60s TTL 内存缓存，避免每次请求读文件
- 文件读取失败返回过期缓存（而非崩溃）
- `process.on('uncaughtException')` + `unhandledRejection` 全局防护
- HTTP handler 包裹 try-catch，异常返回 500 而非进程退出
- `/health` 增加 `cacheAge` 字段用于监控

#### update-ipc.ts 增强
- `httpCheckVersion()`: 3次重试（间隔1s/3s/5s）
- `httpFallbackDownloadWithProbe()`: 新函数 — 5次重试 + URL HEAD预探测
  - 每次重试前重新获取 `/api/latest` 确认版本状态
  - 版本 status != "published" 自动跳过
  - HEAD 探测安装包 URL 确认可达后再下载
- `update:download`: HTTP fallback 从3次提升到5次

## v1.73.79 Auto-Repair 修复 + HTTP fallback 修复 (2026-06-15)

### 问题背景
1. **`loadPrompts is not a function`**: 用户升级到 v1.73.78 后，Quest/Agent 模式初始化时报错
   `Provider initialization failed: (0, import_core4.loadPrompts) is not a function`
2. **安装包文件未找到**: 用户尝试下载更新时提示"安装包文件未找到"

### 根因分析

#### loadPrompts 问题
- `build-main.mjs` 注入的 `__safeRequireCodepilot` (v5) 在模块缺失/损坏时返回 `{}`
- `quest-ipc.ts`/`agent-ipc.ts` 静态 import `loadPrompts`，模块加载时已绑定到 `{}`，`loadPrompts === undefined`
- repair 失败后无兜底，app 白屏不可用
- 附带问题：`main.ts` bootstrap 路径使用 `dirname(process.resourcesPath)` 而非 `process.resourcesPath`

#### 安装包未找到问题
- `/api/latest` 返回相对 URL（如 `/downloads/1.73.79/...`）
- `update-ipc.ts` HTTP fallback 直接传给 Node.js `fetch()`，Node.js fetch 不接受相对 URL

### 修复内容

| 文件 | 修改 | 说明 |
|------|------|------|
| `build-main.mjs` | v5→v6 | 添加 `__stub` 兜底模块（含 noop loadPrompts），修复失败时返回 stub 而非 `{}` |
| `main.ts` | 路径修正 | `dirname(process.resourcesPath)` → `process.resourcesPath` |
| `update-ipc.ts` | URL 补全 | 相对 URL 前加 `https://ide.zhejiangjinmo.com` |
| `electron-builder.json` | output | `release-v17377` → `release-v17379` |
| `core/package.json` | 版本 | `1.73.78` → `1.73.79` |

## v1.73.78 账号系统全面修复 (2026-06-15)

### 问题背景
1. 管理后台用户管理页面无数据、无法添加用户
2. 桌面端IDE新版本需要登录，但无账号可用

### 根因分析

#### 问题1：管理后台 API 路由不匹配
- 管理后台 Vue SPA 使用 `axios.create({baseURL: "/admin/api"})` 
- Nginx 重写规则: `/admin/api/users` → `/api/admin/users`
- 但后端 `admin-api.js` 的 user 路由注册在 `/api/users`（无 admin 路径段）
- 导致管理后台调用 `GET /admin/api/users` 时请求被转发到 `/api/admin/users`，后端无此路由

#### 问题2：桌面端无法登录
- 桌面端本地数据库 `database.ts` 的 schema 中没有 `users` 表
- `auth-service.ts` 使用了 `db.exec()` 但 sql.js 适配器的 `exec()` 返回 void（非返回查询结果集）
- `auth-store.ts` 的自动登录回调 `window.electronAPI.auth.login('admin', 'admin123')` 因无用户且数据库API不兼容而失败

### 修改内容

#### 服务端 (services/backend/admin-api.js)
| 修改 | 说明 |
|------|------|
| 将 login/check-password/change-password 抽为独立处理函数 | 解耦路由注册与业务逻辑 |
| 添加 `/api/login`、`/api/check-default-password`、`/api/change-password` 路由 | 在 `/api/` 路径下挂载（nginx rewrite 后 `/admin/api/*` → `/api/*`）|
| 添加 `/api/admin/users` 路由变体 (GET/POST/PUT/DELETE) | 向后兼容原有 `/api/users` |
| 修复 changePasswordHandler 嵌套 try 语法错误 | 恢复 cloud-server 启动 |

#### Nginx 配置
| 修改 | 说明 |
|------|------|
| `rewrite ^/admin/api/(.*) /api/admin/$1 break` | → `rewrite ^/admin/api/(.*) /api/$1 break` |
| 作用：`/admin/api/users` → `/api/users`（匹配后端实际路由）|

#### 桌面端 (desktop/electron)
| 文件 | 修改 |
|------|------|
| `src/auth/auth-service.ts` | 将所有 `db.exec()` 改为 `db.all()`（sql.js 适配器兼容） |
| 同上 | 添加 `ensureAuthTables()` 函数，自动创建 users/conversations/messages 表 |
| `src/ipc/auth-ipc.ts` | 登录失败且无用户时自动创建默认 admin 用户并重试 |
| 同上 | 导入 `saveDatabase` 以持久化新用户 |
