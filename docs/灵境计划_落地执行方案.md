# 【财务与商业模式架构师】执行方案

## 1. 【单位经济模型】—— 精确到小数点后两位

**第1步：计算单客户LTV（客户生命周期价值）**
- **公式**：LTV = (ARPU × 毛利率) × (客户生命周期)
- **客户生命周期** = 1 / (1 - 续费率)
- **ARPU（每用户季度收入）** = 299元

**三种续费率情景计算**：
1. **悲观（30%续费率）**：
   - 生命周期 = 1 / (1 - 0.3) = 1.43季度 ≈ 4.3个月
   - LTV（毛收入） = 299 × 1.43 = **427.57元**
   - 注：此情景下客户留存差，需紧急优化产品。

2. **中性（50%续费率）**：
   - 生命周期 = 1 / (1 - 0.5) = 2季度 = 6个月
   - LTV（毛收入） = 299 × 2 = **598元**

3. **乐观（70%续费率）**：
   - 生命周期 = 1 / (1 - 0.7) = 3.33季度 ≈ 10个月
   - LTV（毛收入） = 299 × 3.33 = **995.67元**

**第2步：API调用成本估算**
- **核心假设**：小微商家销售场景，每月平均与100个潜在客户互动。
- **单次话术生成**：调用大模型API（如DeepSeek），平均输入+输出共500 tokens。
- **单次成本**：按DeepSeek API 价格（假设0.5元/千tokens），单次成本 = 0.5 * (500/1000) = **0.25元**。
- **每月调用次数**：100个客户 × 每个客户平均生成3次话术 = 300次。
- **每月API成本/用户**：300次 × 0.25元 = **75元**。
- **季度API成本/用户**：75元 × 3 = **225元**。

**第3步：服务器/算力成本分摊**
- **现有硬件**：本地AMD AI Station + 远程Win11 GPU服务器。主要成本为远程服务器租赁。
- **成本估算**：远程96GB GPU服务器（按市场价）约 **4000元/月**。
- **用户分摊**：假设目标服务500个付费用户。
- **每用户每月服务器成本**：4000元 ÷ 500用户 = **8元/月**。
- **每用户季度服务器成本**：8元 × 3 = **24元**。

**第4步：毛利率测算**
- **季度单客户收入**：299元
- **季度单客户可变成本**：API成本(225元) + 服务器分摊(24元) = **249元**
- **单客户季度毛利**：299 - 249 = **50元**
- **毛利率**：50 / 299 = **16.7%**
- **结论**：当前定价下毛利率偏低，高度依赖API成本控制。**必须通过优化提示词、缓存常见回复、使用更便宜模型混合策略，将单次API成本降低30%以上，目标毛利率应提升至30%+**。

## 2. 【定价策略】—— 立即执行的方案

**为什么是299元/季度？**
- 对标市场：小微企业SaaS工具（如微伴、芝麻）客单价在800-1500元/年。299元/季度（约1200元/年）处于中位，试探市场接受度。
- 心理门槛：低于300元，决策压力小。季度付降低首次尝试门槛。
- **必须附加免费试用期**：**立即执行“7天全功能免费试用”**。理由：B端决策需要体验；可收集手机号作为销售线索；7天足以体验核心功能。

**阶梯定价方案（上线首日即实施）**：

## 3. 【财务生存计划】—— 按月烧钱计算

**第1步：按月计算开支明细（未来6个月）**
- **人力成本（核心）**：

**第2步：达到盈亏平衡需要多少付费用户**

- 月可变成本/用户 = API成本(75) + 服务器分摊(8) = 83元

- **现金流测算**：

  - **第4个月**：累计用户300，收入可观，且通过技术优化将单用户API成本降至50元/月，**实现月度现金流转正**。

## 技术架构与实施方案（Kimi / moonshot-v1-32k）

### 1. 从Flutter APP到微信小程序的迁移方案

#### 后端代码复用与重写
- **复用**：FastAPI框架下的API逻辑基本可以复用，因为它提供了RESTful API的支持，与前端的具体实现无关。需要重写的是API的返回格式，以适配微信小程序的数据格式要求。
- **重写**：数据库交互层可能需要根据新的数据库选择进行调整，同时，与Flutter APP特有的交互逻辑相关的代码需要重写。

#### 微信小程序前端技术选型
- **原生开发**：不推荐，因为开发效率低，且后续维护成本高。
- **Taro**：推荐使用，因为它允许使用React的编程模式来开发小程序，且社区活跃，生态丰富。
- **uni-app**：次选，虽然也支持多端开发，但在微信小程序开发方面不如Taro成熟。

#### 迁移工期估算和关键里程碑
- **第1周**：完成需求分析和设计，确定技术选型，搭建开发环境。
- **第2周**：完成API接口的适配和前端页面的原型开发。
- **第3周**：完成前端页面的开发和后端API的联调。
- **第4周**：进行测试和优化，准备上线。

### 2. 后端架构调整

#### FastAPI API路由调整
- **保留**：与业务逻辑紧密相关的API，如用户管理、数据查询等。
- **砍掉**：与C端用户相关的API，如个人资料设置等。
- **新增**：话术生成、客户跟进提醒、成交漏斗看板相关的API。

#### 数据库迁移
- **迁移至MySQL**：因为B端业务通常数据量较大，需要更强的事务性支持和更高的并发处理能力，SQLite不适合。
- **迁移理由**：MySQL具有更好的性能和扩展性，适合商业应用。

#### 记忆系统适配B端场景
- **适配方案**：将个人记忆系统改造为商家客户记忆系统，通过商家ID区分不同的商家数据。
- **实现方式**：在数据库中增加商家ID字段，用于区分不同商家的数据。

### 3. 大模型调用架构

#### 话术生成模型选择
- **模型选择**：使用Sentence-BERT进行语义匹配，结合大模型API生成话术。
- **Prompt设计**：设计针对性的prompt，以适应不同的业务场景和客户需求。

#### API调用成本控制
- **缓存策略**：对于高频调用的话术生成，使用内存缓存（如Redis）减少API调用。
- **模型选择**：根据成本和性能选择适合的模型，平衡响应时间和成本。

#### 本地GPU vs 云端API成本对比
- **成本对比**：本地GPU适合处理高吞吐量的任务，成本固定；云端API按需付费，适合波动性负载。
- **选择建议**：初期以本地GPU为主，待业务稳定后，根据实际负载情况逐步迁移至云端API。

### 4. 部署与运维

#### 生产环境部署方案
- **云服务商**：推荐使用阿里云或腾讯云，选择它们的轻量应用服务器。
- **配置**：至少2核CPU，4GB内存，50GB SSD存储。

#### 监控告警方案
- **工具**：使用Prometheus进行监控，Grafana进行数据可视化。
- **告警**：设置CPU和内存使用率阈值，超过阈值自动告警。

#### 灰度发布策略
- **策略**：使用Feature Flags进行功能开关控制，逐步向用户开放新功能。

## 法律合规与数据安全（Hunyuan / hunyuan-pro）

**合规实施方案**

**一、【数据合规改造清单】**

1. **当前记忆系统存储数据类型及分类**
   - 用户交互记录：包括用户对话内容、行为日志等，部分属于个人信息。
   - 商家信息：包括商家名称、联系方式、交易数据等，属于商业秘密及个人信息。
   - 系统日志：服务器操作记录，部分可能间接涉及个人信息。

2. **合规处理方案**
   - 个人信息：采用脱敏处理，如替换姓名为随机标识，模糊化处理手机号、邮箱等。
   - 敏感个人信息：加密存储，使用AES-256位加密算法。
   - 商业秘密：严格限制访问权限，采用访问控制列表（ACL）进行管理。

3. **技术实现建议**
   - 使用Python的cryptography库进行AES-256加密。
   - 利用正则表达式和字符串替换实现数据脱敏。

**二、【必须准备的法律文件】**

1. **用户服务协议模板要点**
   - 明确服务内容、使用方式、费用及支付方式。
