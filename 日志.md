
## Phase 78: 修复 Preview/Spec/文件变更三大功能 — v1.42.7 (2026-05-19)

### 任务
修复 Quest Mode 中三个从未正常工作的核心功能：
1. **Preview 实时预览** — iframe 加载前端页面进行实时预览
2. **Spec 自动生成** — AI 输出 `:::spec` 块时自动识别并显示在 Spec 标签
3. **文件变更自动处理** — auto 模式下自动接受文件变更

### Bug 1: Preview 功能从未生效

**根因 1a: URL 检测正则过于严格**
`quest-ipc.ts` 中正则 `/https?:\/\/localhost:\d+/` 只匹配 `localhost`，不匹配 `127.0.0.1`、`0.0.0.0` 等常见地址。

**修复**: 增强为 `PREVIEW_URL_RE`，匹配 `localhost`、`127.0.0.1`、`0.0.0.0`、`[::1]`。新增 `extractPreviewUrls()` 函数，含去重逻辑防止重复发送事件。同时为 `quest:resume` 路径也添加了 URL 检测。

**根因 1b: iframe 加载无错误反馈**
`QuestPreviewTab.tsx` 中 `onLoad` 事件未处理加载失败场景。

**修复**: 添加 `loadError` 状态 + 15 秒超时检测 + 错误提示 UI（含重试/关闭按钮）。新增 `127.0.0.1` 快捷 URL。

**根因 1c: Electron webSecurity 可能阻止 iframe**
`main.ts` 中 `webSecurity: true` 可能阻止 `file://` 页面加载 `http://` 内容。

**修复**: 添加 CSP header `frame-src 'self' http://localhost:* http://127.0.0.1:*`。

### Bug 2: Spec 自动生成未生效

**根因**: `detectSpecBlock` 正则 `/:::spec\s*\n([\s\S]*?)\n:::/` 要求闭合 `:::` 在独立行，\s 可能匹配过多换行。Windows `\r\n` 兼容性差。

**修复**: 使用 `[ \t]*\r?\n` 替代 `\s*\n`，避免跨行匹配；添加 `\r?\n` 支持 Windows/Unix 双换行风格；空内容检测跳过；Heading 格式同步增强。

### Bug 3: 文件变更自动处理（设置为自动）未生效

**根因**: `autoMode === 'auto'` 只控制 tool confirmation（bash/MCP 自动执行），但不控制 file_write/file_edit 变更结果的自动接受。所有文件变更始终为 `pending` 状态。

**修复**: 在 `useQuestEvents.ts` 的 `file_snapshot` 事件处理器中，检测当前任务的 `autoMode`。如果为 `'auto'`，自动调用 `acceptFile()` 将变更状态设为 `accepted`。

### Bug 报告
用户在桌面端灵境IDE成功注册账号后，在移动端APP登录时提示"密码账号错误"，且管理后台看不到注册的用户。

### 根因分析（4个Bug）

| # | Bug | 根因 | 影响 |
|:-:|:----|:-----|:-----|
| 1 | **密码验证失败** | signup 处理器将 `hashPassword()` 返回的 `salt:hash` 拆分为 salt 存 `password_salt`、hash 存 `password_hash`。但 `verifyPassword()` 期望 `password_hash` 存储完整 `salt:hash` 字符串 → `stored.split(':')` 时只有1个元素，salt 丢失 → 验证始终返回 false | **所有登录失败** |
| 2 | **重复 signup 路由** | server.js 有2个 `app.post('/api/auth/signup')` 处理器(第600行和685行)，第二个虽被覆盖但仍存在 | 代码混乱、维护困难 |
| 3 | **CloudAuthClient 字段名错误** | 桌面端 `CloudAuthClient.login()` 发送 `{ email, password }`，但服务器期望 `{ username, password }` | 桌面端部分登录路径会失败 |
| 4 | **email 空值 UNIQUE 冲突** | 移动端注册时 `email: ''`，`INSERT ... VALUES (..., email || '', ...)` 插入空字符串，`email TEXT UNIQUE NOT NULL` 导致只有第1个无邮箱用户能注册成功 | 多个用户注册失败 |

### 修复方案

**server.js 修改：**

1. **signup 处理器**：`INSERT` 改为存储完整 `salt:hash` 到 `password_hash`，同时保留 `password_salt` 列兼容旧数据
2. **删除重复 signup 路由**：删除第685-732行的第2个 `/api/auth/signup` 处理器
3. **verifyPassword 函数**：支持两种存储格式（NEW: salt:hash 合并 / OLD: 独立 salt+hash）
4. **login 查询**：`SELECT ... password_salt FROM users`，传递给 verifyPassword
5. **email 空值处理**：`email || (username + '@lingjing.local')` 确保每个用户有唯一邮箱

**CloudAuthClient.ts 修改：**
- `{ email, password }` → `{ username: email, password }`

| 文件 | 变更 |
|:-----|:------|
| `cloud-server/server.js` | signup INSERT、删除重复路由、verifyPassword、login 查询、email 空值 |
| `packages/renderer/src/auth/CloudAuthClient.ts` | login 字段名 `email` → `username: email` |

## Phase 67: v1.42.4 — 登录页Input文字颜色修复 + 移动端APK签名修复 (2026-05-18)

### 问题 1: 登录/注册页输入框文字不可见
用户反馈灵境AI编程助手登录/注册页面输入框（用户名、邮箱、密码）的背景色与文字颜色相同，无法看到输入内容。

### 根因
`packages/renderer/src/components/ui/Input.tsx` 中 input 文字使用 `text-neutral-100`。在 `tailwind.config.js` 中 `neutral-100` 映射到 CSS 变量 `--cp-tab-inactive`，暗色模式下 `--cp-tab-inactive` = `#2d2d2d`。背景 `bg-neutral-800` 映射到 `--cp-sidebar` = `#252526`。两者几乎相同 (#2d2d2d ≈ #252526) → 文字不可见。

### 修复
`Input.tsx`: `text-neutral-100` → `text-cp-text`（使用 `--cp-text` 变量，暗色模式 = `#ffffff`，浅色模式 = `#1a1a1a`）

### 问题 2: 移动端 Android APK 安装卡在 85%
用户反馈 Android 安装灵境 APK 时一直卡在约 85% 无法完成安装。

### 根因
GitHub Actions 工作流中 `./gradlew assembleRelease` 构建时**没有配置 release signingConfig**。生成的 APK 未使用 release keystore 签名，Android 的包管理器在签名验证阶段挂起（约 85%），无法完成安装。

### 修复
`.github/workflows/build-apk.yml`:
1. 新增 `Setup release keystore` 步骤 — 使用 `keytool -genkey` 生成 release keystore
2. 创建 `keystore.properties` 供 Gradle 读取
3. 新增 `Patch build.gradle for release signing` 步骤 — 注入 signingConfigs.release 配置
4. 新增 `Verify APK signing` 步骤 — 使用 apksigner/jarsigner 验证签名
